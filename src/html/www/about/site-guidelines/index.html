Site guidelines

Why?

Why not? This website began as an attempt to learn the basics of web development, starting from a single HTML file. As this site grows in size and complexity, I aim to continue learning the intricacies of front-end development without the abstractions provided by a framework or library. These guidelines are set in place not only to keep things interesting for myself, but also to allow me to develop some best practices through good ole trial-and-error. If any of the guidelines are broken in any way, please file an issue here.

The guidelines

1. JavaScript is "extra"

JavaScript will be used for enhancements only (e.g., AJAX). JavaScript is technically an optional feature, despite its usefullness, as many browsers allow you to disable JavaScript (though few users actually do so). Anything considered "necessary" for the site to work should not rely on JavaScript being available.

2. Performance is desirable

Techniques such as gzipping and minification will be experimented with in an attempt to reduce file sizes (and help traffic performance). Also, I will work on making sure that stylesheets and script files are properly optimized, to ensure that an i7 + GTX 1080 rig isn't needed for visiting this page. Any optimization suggestions are welcome.

3. Accessibility is important

Accounting for visual impairments and other disabilities changes how a website can be structured, along with how it is styled and presented. I will do my best to conform to W3Câ€™s web content accessibility guidelines (WCAG 2.0) during the development of this site, even when implementing fancier interfaces such as modal windows or navigation menus.

4. No WCMS

Web Content Management Systems, such as WordPress, are not going to be used. While they are extremely useful tools, and should probably be used for anything serious, I am choosing to do most things manually for now. Along with it being a learning experience, this development workflow will give me a better appreciation for WCMS-type tools in the future.
